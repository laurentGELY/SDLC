Modèle SDLC v1.3 · Mis à jour le 03/06/2026
Vue d'ensemble

Concepts clés

🏗️
/sdlc-init

Bootstrappe un repo vide : crée toute la structure de gouvernance en une commande.

🔄
/sdlc-sync

Aligne un projet existant sur le modèle courant sans écraser son tuning local.

📋
Marqueur de version

Ligne <!-- SDLC version : vX.Y --> dans Claude.md et STANDARDS.md. Son absence signifie que le projet est antérieur au modèle générique.

⬆️
Circuit bidirectionnel

Le tuning d'un projet peut remonter vers le modèle via [SDLC_CANDIDATE] dans LESSONS_LEARNED.md.

ℹ️
Règle fondamentale

Le tuning local d'un projet prime toujours sur le modèle générique. Le SDLC-Sync apporte ce qui manquait — il n'écrase pas ce qui existe.

Détecter la situation

Avant toute opération, une commande détermine quelle action entreprendre :

bash 
grep "SDLC version" Claude.md STANDARDS.md 2>/dev/null || echo "ABSENT"
Résultat Situation Action
Aucun fichier trouvé Repo vide ou sans gouvernance → Initialiser avec sdlc-init.sh
ABSENT Gouvernance antérieure au modèle générique → Synchroniser avec /sdlc-sync (delta complet)
SDLC version : vX.Y Projet versionné, potentiellement en retard → Synchroniser avec /sdlc-sync (delta vX.Y → courant)
Version = version courante Projet à jour Rien à faire ✓
Opération 1

Initialiser un nouveau projet

⚠️
Prérequis

Repo git initialisé (git init fait). Répertoire courant = racine du nouveau projet. Le repo SDLC est accessible en chemin absolu ou cloné localement.

Étape 1 — Lancer le script

bash 
# Depuis la racine du nouveau projet
bash /chemin/vers/sdlc-toolkit/sdlc-init.sh "Nom du projet"

Le script crée la structure complète, remplace les placeholders mécaniques ([Nom du projet], date), et dépose le skill /sdlc-sync dans .claude/skills/.

Étape 2 — Ouvrir Claude Code et compléter

Le script fait le travail mécanique. Claude Code fait le travail de sens. Ouvrir une session Claude Code dans le repo et demander :

Prompt Claude Code 
Le bootstrap mécanique est fait. Complète la gouvernance :
- §Rôle dans Claude.md : [décrire le domaine du projet]
- §Limites bash : [lister les actions impossibles en bash]
- §Démarrage §2 : commandes d'état spécifiques au projet
- §Carte des étapes dans STANDARDS.md : vider, sera rempli sprint 1
- specs/SPEC.md : structure vide du domaine (§Vue d'ensemble · §Architecture · §Modules)
- .claude/skills/diagnostic/SKILL.md : commandes de diagnostic du système
- Décider des sections [ACTIVER si…] dans pre-tool-bash.sh

Grep de validation final :
grep "\[→ ADAPTER\]\|\[Nom du projet\]\|\[ACTIVER" Claude.md STANDARDS.md .claude/hooks/pre-tool-bash.sh
→ zéro résultat attendu avant le commit.

Étape 3 — Vérifier et committer

bash 
# Vérification finale
grep "SDLC version" Claude.md STANDARDS.md   # doit afficher vX.Y
grep "\[→ ADAPTER\]" Claude.md STANDARDS.md  # doit être vide
echo '{"tool":"bash","input":{"command":"echo ok"}}' \
  | bash .claude/hooks/pre-tool-bash.sh       # doit retourner exit 0

# Commit
git add -A
git diff --stat HEAD  # vérifier la liste
git commit -m "chore: bootstrap SDLC v1.3"
✅
Critères d'acceptation

Marqueur SDLC version présent · zéro placeholder résiduel · hook smoke test exit 0 · CHANGELOG.md avec entrée Sprint 0 · doc/ROADMAP.md avec Sprint 1 en §Now.

Opération 2

Mettre à jour un projet existant

Le skill /sdlc-sync est exécuté par Claude Code dans le projet cible. Il suit quatre étapes : inventaire → tri → application (après aval) → traçabilité.

/sdlc-sync
→
A · Inventaire
→
B · Tri
→
⏸ Aval humain
→
C · Application
→
D · D-SYNC-XX

Lancer le skill

Dans Claude Code (projet cible) 
/sdlc-sync

Règle de tri — ce que fait Claude à l'étape B

SituationAction
Section absente, universelle → Ajouter
Section absente, conditionnelle → Vérifier la contrainte dans le projet, ajouter si pertinent
Section présente, formulée différemment → Comparer : migrer si bénéfice net, sinon laisser
Tuning local sans équivalent SDLC → Laisser intact — noter comme candidat remontée
ℹ️
Projet sans marqueur de version

Projet antérieur au modèle générique. Claude traite toutes les sections comme "manquantes" et applique la règle de tri sur l'intégralité du modèle. Le tuning local existant est toujours préservé.

Output attendu à l'étape D

Une entrée D-SYNC-XX dans doc/DECISIONS.md du projet, et le marqueur de version mis à jour dans Claude.md et STANDARDS.md.

Format D-SYNC-XX 
## D-SYNC-01 · Alignement SDLC vX.Y → vZ.W · [date]

Appliqués    : [liste des sections ajoutées/migrées]
Ignorés      : [liste + raison]
Laissés      : [tuning local conservé]
Remontées ?  : [candidats SDLC_CANDIDATE si trouvés]
Opération 3

Faire évoluer le modèle SDLC

Le modèle évolue par deux chemins : une décision directe sur le toolkit, ou une remontée terrain depuis un projet.

1
Identifier la source

Décision directe — on identifie un manque dans le modèle indépendamment de tout projet.

Remontée terrain — un [SDLC_CANDIDATE] dans LESSONS_LEARNED.md d'un projet cible a été synthétisé par /retrospective.

2
Vérifier les invariants

Lire 00-README.md §3 avant toute modification. Les 4 invariants (vérification exécutable, circuit fermé, contexte chirurgical, boucle de rétroaction) doivent être préservés ou la violation explicitement justifiée.

3
Modifier les templates concernés

Mettre à jour le numéro de version dans l'en-tête de chaque fichier modifié.

Grep de validation post-modification
grep "\[→ ADAPTER\]" 01-Claude-md-TEMPLATE.md  # vérifier cohérence
4
Documenter dans 07-DECISIONS-SDLC.md

Nouvelle entrée M-XXXX-NN avec décision retenue, alternative écartée, justification. Ajouter la décision au tableau de compatibilité avec sa qualification universelle / conditionnelle.

5
Mettre à jour 00-README.md

Incrémenter la version dans §Historique des versions. Cocher la checklist §4.

6
Sync Claude.ai + commit

Remplacer les fichiers modifiés dans le projet Claude.ai "Modèle SDLC". Committer avec docs(sdlc): description · vX.Y → vZ.W.

⚠️
Après une évolution du modèle

Tous les projets avec un marqueur de version antérieur à la nouvelle version sont désormais candidats à un /sdlc-sync. Ce n'est pas automatique — c'est une décision humaine au cas par cas.

Note sur /sdlc-init

Ce skill n'existe pas dans un repo vide — il est déposé par sdlc-init.sh lors du bootstrap. Le script est le vrai point d'entrée pour un nouveau projet.
